# 中期答辩问答预测手册

> 适用场景：计算机毕设中期答辩，展示"基于深度学习的代谢组学批次效应系统"时的常见提问及参考答案。  
> 指导老师可能对生物学背景不熟悉，以计算机系统与算法视角为主。

---

## 一、关于"为什么做这个"

### Q1：你这个系统和 MetaboAnalyst 有什么区别，为什么不直接用现有工具？

**参考答案：**

MetaboAnalyst 是通用型代谢组学在线分析平台，存在三点不足：
1. **没有深度学习缺失值填充**，仅支持 mean/median/KNN 等传统策略；
2. **批次效应与缺失值是分步处理**，误差会逐步累积；
3. **没有知识图谱溯源**，差异代谢物无法回溯到通路-反应-酶的关联关系。

本系统聚焦批次效应处理这一核心场景，引入了 Autoencoder 深度学习填充，并提供可量化的方法对比评估框架，是对现有工具的补充与改进。

---

### Q2：你这个"批次效应"用中文怎么解释？普通人能理解吗？

**参考答案：**

不同时间、不同仪器做同一批实验，测出来的数值会系统性地偏高或偏低——这种偏差不是生物本身的差异，而是实验误差，叫做"批次效应"。

打个比方：两台体重秤有固定的示数偏差，你需要先"校准"再比较。批次效应校正就是做这个校准的过程。

---

### Q3：为什么是代谢组学？不做图像或者 NLP 之类更主流的方向？

**参考答案：**

代谢组学是近年来生命科学领域的研究热点，数据特点是**高维度（上千个特征）、小样本（几十到几百个样本）、多批次**，这与图像/NLP 场景有本质差别，现有通用深度学习框架无法直接套用。

从计算机角度看，这个场景对数据预处理和特征工程的要求更高，批次效应校正是真实科研痛点。系统的价值在于把复杂的算法流程封装成可用的 Web 工具，降低生物研究人员的技术门槛。

## 二、技术选型（高频考点）

### Q4：后端为什么选 FastAPI，不用 Django 或者 Flask？

**参考答案：**

三者对比如下：

| 框架 | 优点 | 不选择的原因 |
|------|------|------------|
| **Django** | 功能全、生态成熟 | 偏 MVC 全栈框架，本项目前后端分离，Django 大量内置功能（模板、ORM Admin）用不上，偏重 |
| **Flask** | 轻量、灵活 | 没有内置参数校验和自动文档，需要额外集成 Marshmallow/Pydantic，工作量大 |
| **FastAPI** ✅ | 基于 Pydantic 自动参数校验、内置 OpenAPI 文档、原生支持 async | 与本项目需求匹配：纯 API 服务、参数校验严格、开发效率高 |

FastAPI 还自带 `/docs` Swagger 界面，调试接口非常方便，在开发阶段节省了大量时间。

- **选 Vue3 + TypeScript**：类型安全可以提前发现接口字段不匹配的问题，配合 Vite 构建速度快，Element Plus 组件库 UI 质量高，ECharts 集成简单。

**参考答案：**

本系统定位是**单机/实验室内网部署**的科研辅助工具，不是高并发生产服务：

### Q8：为什么用 PyTorch 而不是 TensorFlow 做深度学习？

**参考答案：**

- **动态计算图**：PyTorch 的 eager execution 更直观，调试方便，适合研究性项目快速迭代；
- **社区趋势**：近年来学术界主流已转向 PyTorch，相关论文代码复现更容易找到参考；
- **轻量部署**：本项目的 Autoencoder 是 CPU 推断即可完成的小模型，PyTorch 在这个规模下性能完全足够；
- **vs TensorFlow**：TF2 的 Keras API 虽然简单，但细粒度控制（如自定义 Masked Loss）不如 PyTorch 直接。

## 三、关于深度学习（必问，是题目核心）

### Q10：你用了什么深度学习模型？为什么选这个？

**参考答案：**

使用了 **Autoencoder（自动编码器）**，由两部分组成：
- **Encoder**：将样本的高维特征（1000+ 个代谢物）压缩到低维潜空间（64 维），学习整体数据分布；
- **Decoder**：从低维潜空间重建回原始维度。

选择 Autoencoder 的原因：
1. 代谢组学数据维度高、样本数少，适合无监督表示学习；
2. 训练策略采用 **Masked Reconstruction**，仅对已知值位置计算 MSE 损失，让模型从周围特征的模式推断缺失位置，天然适合缺失值填充任务；
3. 参考了 Gondara & Wang 2018 年的 MIDA（Multiple Imputation using Denoising Autoencoders）框架。

---

### Q11：为什么不用 Transformer 或者 GAN？现在这些更流行。

**参考答案：**

三种方案对比：

| 模型 | 不选的原因 |
|------|----------|
| **Transformer** | 自注意力机制对序列/长程依赖建模有优势，但代谢组学特征之间不存在序列关系，位置编码无意义；且样本数少时 Transformer 容易过拟合 |
| **GAN** | 训练不稳定（模式崩溃问题），需要大量样本和调参经验，代谢组学小样本场景不适合；GAN 主要用于数据增强，不直接输出缺失值估计 |
| **Autoencoder** ✅ | 架构简单、训练稳定、在小样本高维数据上表现可靠，Masked Reconstruction 策略天然契合缺失值填充任务 |

选择 Autoencoder 是在**模型复杂度、训练稳定性、数据规模适配性**之间做出的合理权衡。

---

### Q12：深度学习在这里具体解决什么问题？效果怎么样？

**参考答案：**

解决**缺失值填充**问题。

评估框架是 **Mask-then-Impute**：
1. 在已完整填充的矩阵上，随机遮蔽 15% 的值（模拟真实缺失）；
2. 分别用 mean / median / KNN / Autoencoder 四种方法填充遮蔽位置；
3. 与真实值对比，计算 RMSE / MAE / NRMSE 三项指标；
4. 重复多次取均值，确保统计稳定性。

结果可在系统"缺失值填充方法评估"区块实时查看，最优方法自动标注。

---

### Q13：你的 Autoencoder 是自己写的还是调库？训练细节是什么？

**参考答案：**

核心模型自主实现（PyTorch），架构为：

```
Input(n_features) → Linear(256) → ReLU → BatchNorm → Dropout(0.1)
                  → Linear(64)  → ReLU                              ← 潜空间
                  → Linear(256) → ReLU → BatchNorm
                  → Linear(n_features)                              ← 重建输出
```

训练细节：
- **损失函数**：仅在已知值位置计算 MSE，公式为 `loss = ((recon - x)² × obs_mask).sum() / obs_mask.sum()`；
- **优化器**：Adam（lr=1e-3，weight_decay=1e-5）；
- **学习率调度**：CosineAnnealingLR，训练结束时 lr 衰减到初始值的 10%；
- **训练轮数**：80 epochs；
- **缺失位置初始化**：用列均值填充，为网络提供合理初始输入，而非全 0。

---

### Q14：你的深度学习比 KNN 好多少？有没有可能不如传统方法？

**参考答案：**

在代谢组学小样本场景（几十到几百个样本），Autoencoder 对 KNN 的改进幅度不一定稳定。

但这正是系统建立**可量化评估框架**的价值所在：
- 系统并不强制使用某种方法，而是让用户看到各方法的 RMSE 对比，自主选择最优方案；
- 对于不同数据集，最优方法可能不同；
- Autoencoder 在特征间具有复杂相关结构时通常优于 KNN，但在稀疏场景下可能不如均值填充。

这种"方法对比 + 用户决策"的设计思路本身就是系统的工程价值。

## 四、关于批次效应校正

### Q15：你实现了几种批次效应校正方法？有什么区别？

**参考答案：**

实现了两种方法：

| 方法 | 原理 | 特点 |
|------|------|------|
| **Baseline（逐特征位置尺度对齐）** | 对每个特征，将各批次的均值/标准差归一化到全局分布 | 简单高效，无需生物学协变量，自主实现 |
| **ComBat（neuroCombat，经验 Bayes）** | 用参数先验对批次参数做收缩估计，可传入生物学协变量加以保护 | 领域标准方法，参考 Johnson et al., 2007 |

两种方法均在系统中提供可视化对比，用户可根据评估指标选择。

---

### Q16：ComBat 是什么？你是自己实现的吗？

**参考答案：**

ComBat 是批次效应校正领域最经典的方法，由 Johnson et al. 2007 年在 Biostatistics 期刊发表。核心思想是：

1. 对每个特征建立线性模型，将观测值分解为"全局均值 + 批次效应 + 生物学效应 + 残差"；
2. 用**经验 Bayes（Empirical Bayes）**方法估计批次参数，通过引入先验约束降低小样本场景的估计方差；
3. 将批次效应从原始数据中减去，保留生物学效应。

本项目**封装了 neuroCombat 库**（Python 实现），并自主编写了：
- 输入校验（批次数、每批次样本数）
- 生物学协变量传入接口
- 安全封装（`run_combat_safe`），即使 ComBat 失败也不中断整体流程

这是工程化封装，不是"直接调库"，体现了对算法的理解。

---

### Q17：校正完批次效应之后，怎么知道效果好不好？

**参考答案：**

用三个量化指标评估，在系统结果页面直接展示校正前后的数值对比：

1. **批次质心距离（batch_centroid_separation）**：PCA 空间中各批次样本质心之间的平均欧氏距离。越小说明各批次整体分布中心越接近，批次混合越好；
2. **Silhouette(batch_id)**：以批次标签计算的轮廓系数。越低（越负）说明批次间样本点难以区分，批次效应被削弱；
3. **Silhouette(group_label)**：以生物分组标签计算的轮廓系数。越高说明生物学分组结构得到保留，真实生物差异未被破坏。

同时提供 **PCA 散点图前后对比**，直观观察批次聚类是否被消除。

---

### Q18：为什么要同时关注 Silhouette(batch_id) 和 Silhouette(group_label)？只看一个不行吗？

**参考答案：**

批次效应校正存在一个经典矛盾：

- 如果**过度校正**，可能把真实的生物学差异也一起消除了——此时 `Silhouette(batch_id)` 会很低（好），但 `Silhouette(group_label)` 也会下降（差）；
- 如果**校正不足**，批次还是可以区分——此时 `Silhouette(batch_id)` 还是很高（差）。

因此必须**同时监控两个指标**，才能判断方法是否在"去除批次效应"和"保留生物信号"之间取得了良好平衡。这也是为什么系统会同时展示这两个指标的前后变化。

## 五、关于系统架构（计算机角度）

### Q19：前后端是怎么分离的？用了什么技术栈？

**参考答案：**

- **后端**：Python + FastAPI，负责文件管理、任务调度、算法调用，以 RESTful API 形式返回 JSON 结果；
- **前端**：Vue3 + TypeScript + Element Plus + ECharts，负责参数配置、任务提交、结果可视化；
- **数据库**：SQLite（存储任务元数据），矩阵数据以 CSV 文件存储在文件系统；
- **算法层**：独立 `algorithms/` 目录，按功能划分为 `preprocessing/`、`imputation/`、`batch_correction/`、`evaluation/`、`downstream/` 五个子模块，便于扩展。

### Q20：系统的整体数据流是怎么走的？

**参考答案：**

```
用户上传 CSV
    ↓
ImportView → POST /api/upload → 文件存储 + Dataset 记录入库
    ↓
TaskConfigView → POST /api/tasks/{id}/preprocess
              → POST /api/tasks/{id}/impute
              → POST /api/tasks/{id}/batch-correct
    ↓（每步结果写临时 CSV 文件，下一步读取上一步产物）
ResultDashboardView → GET /api/benchmark/... 读取预计算产物 JSON
                   → GET /api/dataset/{name}/... 读取各数据集结果
```

每个处理步骤的中间产物以文件形式持久化，即使后端重启也不会丢失计算结果。

---

### Q21：任务是同步还是异步执行的？大数据量怎么处理？

**参考答案：**

当前主流程（预处理 → 填充 → 批次校正）采用同步 HTTP 接口顺序调用，结果逐步写入临时文件。

Autoencoder 训练耗时较长（代谢组学矩阵需要数分钟），因此放在**离线 Pipeline 中预先运行**，结果直接读取展示，而非用户触发时实时训练。这是合理的工程权衡，避免 HTTP 请求超时。

如需扩展为异步：可引入 Celery 任务队列 + Redis Broker，当前架构预留了任务状态字段（status 字段）和 task_id 体系，改造成本较低。

### Q22：数据库用的什么，存了什么？为什么不把数据矩阵也存进去？

**参考答案：**

使用 SQLite，通过 SQLAlchemy ORM 访问，存储三类元数据：
- **Task 表**：task_id、状态（preprocess_done / impute_done / ...）、参数配置 JSON；
- **Dataset 表**：数据集路径、格式、列名映射（sample/feature/batch/group 列）；
- **Result 表**：结果文件路径、metrics JSON。

矩阵数据不入库的原因：
- 代谢组学矩阵可达数十 MB（1715 样本 × 1180 特征 × float64），频繁读写数据库会严重拖慢性能；
- 矩阵数据是"流水线中间产物"，生命周期与任务绑定，文件系统管理更自然；
- 数据库适合管理结构化元数据和索引，不适合存储大体积二维数值数组。

### Q23：你的系统是怎么组织代码结构的？有什么设计原则？

**参考答案：**

后端采用**分层架构**：

```
api/routes/     ← HTTP 接口层（只处理请求/响应，不含业务逻辑）
services/       ← 业务逻辑层（流程编排、数据读写、算法调用）
algorithms/     ← 算法层（纯计算，无 IO、无数据库依赖）
repositories/   ← 数据访问层（数据库 CRUD，隔离 ORM 细节）
models/         ← 数据库模型
schemas/        ← 请求/响应数据格式（Pydantic）
```

这种分层设计的好处：
1. **算法层可独立测试**，不依赖 Web 框架；
2. **services 层可复用**，既可以被 HTTP 接口调用，也可以被 CLI 脚本调用；
3. **替换任何一层**对其他层影响最小（例如从 SQLite 换 PostgreSQL 只改 repositories）。

## 六、关于算法原理（被追问时）

### Q24：PCA 是什么？你的系统里用它做什么？

**参考答案：**

PCA（主成分分析）是一种降维方法，将高维数据投影到方差最大的方向（主成分）上，用少数几个维度捕捉数据的主要变化。

本系统用 PCA 做两件事：
1. **可视化**：把 1000+ 维的代谢组学矩阵降到 2 维（PC1、PC2），在散点图上展示样本分布，直观观察批次聚类是否存在及校正前后的变化；
2. **指标计算**：在 PC1-PC2 空间上计算批次质心距离和 Silhouette 系数，量化批次效应大小。

PCA 本身不做校正，只是**评估和可视化**的工具。

---

### Q25：Silhouette 系数是什么原理？

**参考答案：**

对于每个样本点 i，Silhouette 系数 s(i) 定义为：

```
s(i) = (b(i) - a(i)) / max(a(i), b(i))

a(i) = 样本 i 到同簇其他样本的平均距离（簇内紧密度）
b(i) = 样本 i 到最近异簇样本的平均距离（簇间分离度）
```

- s(i) 趋向 +1：点在本簇中心，远离其他簇 → 聚类清晰；
- s(i) 趋向 -1：点离其他簇比自己所在簇更近 → 分配错误；
- s(i) 趋向 0：点在两簇边界上。

**本系统的应用**：
- 以 batch_id 为"簇标签"计算 Silhouette，值越低说明批次不可区分（校正好）；
- 以 group_label 为"簇标签"计算 Silhouette，值越高说明生物分组保留（生物信号不丢失）。

---

### Q26：t-test 和 BH-FDR 是什么？为什么要做多重检验校正？

**参考答案：**

- **t-test（独立双样本 t 检验）**：检验两组样本的均值是否有显著差异，得到 p-value。p-value 越小说明差异越不可能是随机造成的；
- **log2 Fold Change**：衡量差异方向和大小，正值表示 group2 上调，负值表示下调；
- **BH-FDR（Benjamini-Hochberg 假阳性率校正）**：

代谢组学数据有 1000+ 个特征，同时做 1000 次假设检验。如果每次检验的显著性阈值是 p<0.05，即使特征没有真实差异，平均也会有 50 个假阳性。BH-FDR 通过调整每个特征的 q-value 控制整体假阳性率，让结论更可靠。

---

### Q27：超几何检验是什么？为什么用它做通路富集？

**参考答案：**

超几何检验（等价 Fisher's exact test）：检验"在差异代谢物集合中，属于某通路的代谢物数量是否显著超出随机期望"。

类比：一个箱子里有 100 个球（全部代谢物），其中 10 个是红球（属于通路 A）。你随机抽 20 个球（差异代谢物），发现其中 8 个是红球。超几何检验回答的问题是：这种情况是随机发生的可能性有多低？

- 如果 p-value 很小，说明差异代谢物显著富集在该通路，该通路可能与研究的生物现象相关；
- 同样需要做 BH-FDR 多重校正，因为要同时检验几百条通路。

---

## 七、关于下游分析

### Q28：通路分析是怎么做的？KEGG 数据是哪里来的？

**参考答案：**

1. 首先对批次校正后矩阵做**差异代谢物分析**：独立双样本 t-test + BH-FDR 多重校正 + log2 Fold Change；
2. 筛选显著上调/下调代谢物，获取对应的 **KEGG Compound ID**；
3. 以全部含注释特征为背景集，做**超几何检验**（等价 Fisher's exact test），识别富集通路；
4. KEGG 数据通过 `rest.kegg.jp` REST API 在线获取，首次运行自动下载并本地缓存，避免重复请求。

结果以**气泡图**（横轴富集倍数、纵轴 -log10 q-value、气泡大小表示代谢物数）和**力导向网络图**（代谢物-通路关联网络）两种形式展示。

---

### Q29：知识图谱是你自己构建的吗？数据从哪里来？

**参考答案：**

基于 **MetaKG 多库整合知识图谱**（整合 KEGG / SMPDB / HMDB 三个数据库），提取与本项目代谢物相关的**一跳关联子图**（代谢物-通路-反应-酶-基因-疾病等节点类型）。

前端用 **ECharts 力导向图**渲染，支持：
- 节点拖拽交互；
- 关键词高亮搜索；
- 按节点类型与关系类型动态过滤。

对于无法做知识图谱的数据集（如 AMIDE，6461 个匿名质谱信号，无代谢物名称），系统给出明确提示，不强行渲染。

## 八、关于数据集

### Q30：你用了哪些数据集？数据是哪里来的？

**参考答案：**

系统集成了四个数据集：

| 数据集 | 规模 | 说明 |
|--------|------|------|
| **Benchmark** | 1715 样本 × 1180 特征 × 7 批次 | 7 个实验批次合并，是批次效应校正的主要验证集 |
| **BioHeart** | 多样本 × 53 代谢物 | 心血管疾病相关代谢组学公开数据集 |
| **MI** | 多样本 × 14 代谢物 | 心肌梗死相关代谢组学数据 |
| **AMIDE** | 多样本 × 6461 匿名特征 | 质谱信号，特征无名称，不支持通路分析和知识图谱 |

数据来源为公开数据集及实验室合作数据，原始数据以 xlsx/csv 格式存储，系统提供完整的导入和预处理 pipeline。

### Q31：为什么支持这么多数据集？有什么意义？

**参考答案：**

多数据集支持体现了系统的**通用性**：

1. **Benchmark（7批次）**：数据量大、批次多，用于验证批次效应校正的主链路效果，也是方法对比实验（baseline vs ComBat）的主要场景；
2. **BioHeart / MI**：特征数少（14-53个），有明确代谢物名称，用于验证差异分析→通路富集→知识图谱溯源这条下游链路；
3. **AMIDE**：特征多（6461个匿名信号），用于验证系统在无注释场景下的降级处理能力（给出明确提示而非报错）。

四个数据集覆盖了"大批次/小特征/多特征/无注释"四种典型场景，证明系统不是只针对单一数据设计的。

## 九、刁难性问题

### Q32：你的 Autoencoder 用了多少层？为什么是这个结构？

**参考答案：**

Encoder：输入层 → Linear(256) → ReLU → BatchNorm → Dropout(0.1) → Linear(64) → ReLU  
Decoder：Linear(256) → ReLU → BatchNorm → Linear(n_features)  

选择这个结构的理由：
- **两层 Encoder**：一层直接从输入维度压缩到 64 维容量不足（特征间关系复杂），三层以上在小样本代谢组学数据上容易过拟合；
- **BatchNorm**：稳定训练，减少内部协变量偏移，对小批次训练效果好；
- **Dropout(0.1)**：轻量正则化，防止过拟合（代谢组学样本数少）；
- **潜空间 64 维**：经验值，对于 1000+ 个特征，64 维能保留足够结构信息又不至于过度压缩。

---

### Q33：你的系统安全性怎么保证？用户上传的数据会不会泄露？

**参考答案：**

当前为本地/局域网部署场景，数据完全存储在本地服务器文件系统，不上传任何第三方服务（KEGG API 只上传化合物 ID 进行查询，不涉及原始数据）。如需生产化部署，可增加 JWT 认证、用户数据隔离和 HTTPS 加密，当前架构预留了扩展空间。

---

### Q34：系统有哪些不足，后续如何改进？

**参考答案（诚实作答，展示思考深度）：**

1. **Autoencoder 未集成到 Web 在线任务流**中，当前只在离线评估 pipeline 中运行，后续可通过异步任务队列实现在线训练；
2. **批次效应校正方法**目前只有 baseline 和 ComBat 两种，可扩展 Harmony、BBKNN 等方法；
3. **多数据集支持**目前依赖手动运行 CLI 脚本生成产物，后续可实现 Web 端一键导入和触发；
4. **评估报告**目前以页面展示为主，可增加 PDF 报告自动生成功能；
5. **缺少用户认证**，当前不支持多用户隔离，适合单人或实验室内网使用。

---

### Q35：你做这个项目，最大的技术难点是什么？

**参考答案（展示真实思考）：**

有两个难点：

**第一个是数据格式的归一化**。四个数据集的原始格式完全不同（有的是特征×样本宽表，有的是样本元数据+特征数据分离），需要设计统一的 Pipeline 接口，让后续所有算法都能接受标准化输入。

**第二个是评估指标的设计**。批次效应校正的好坏很难用单一指标衡量——去除批次的同时可能破坏生物信号。设计"双指标同时监控"（Silhouette(batch_id) 越低 + Silhouette(group_label) 越高）这个评估框架，需要对问题有清晰的理解，不是直接套现有工具能解决的。
